# Design Thinking

## Domain
Product & Innovation

## Description
Human-centered innovation framework using five iterative phases — Empathize, Define, Ideate, Prototype, Test — to solve complex problems by deeply understanding user needs before generating and validating solutions.

## Purpose
Design Thinking prevents teams from committing to solutions before the underlying user problem is well understood. It structures the movement from ambiguous problem spaces to testable solution concepts through disciplined divergence and convergence. By centering the process on observed user behavior rather than assumed needs, it reduces the risk of building products or services that fail to create genuine value.

## When to Use
- The problem is poorly defined or stakeholders hold conflicting assumptions about user needs
- Previous solution attempts have failed to achieve adoption despite technical correctness
- A product or service redesign requires grounding in current user experience before ideation begins
- The innovation challenge involves significant human behavior, workflow, or experience components

## When NOT to Use
- The problem is well-defined, the solution is known, and execution speed is the primary constraint
- The target user population is inaccessible for research within the engagement timeline
- Regulatory or technical constraints eliminate most solution degrees of freedom before ideation
- The organization requires a quantitative feasibility analysis rather than solution generation

## Inputs Required
- A defined innovation challenge or problem space to investigate
- Access to target users for qualitative research (interviews, observation, shadowing)
- Stakeholder alignment on the scope and boundaries of the problem
- Time allocation for iterative prototyping and testing cycles

## Optional Inputs
- Existing customer research, journey maps, or support data to accelerate the Empathize phase
- Competitor or analogous industry examples to seed the Ideate phase

## Diagnostic Process
1. Empathize: conduct user research through interviews, contextual observation, and shadowing to capture genuine user behaviors, motivations, and pain points without imposing pre-existing assumptions.
2. Define: synthesize research data using affinity mapping or insight clustering to produce a specific, actionable problem statement (point of view or "How Might We" framing) that represents a real, validated user need.
3. Ideate: facilitate structured brainstorming sessions to generate a broad set of solution concepts without premature filtering; aim for quantity and diversity before evaluating feasibility.
4. Prototype: select the top two to three solution concepts and build low-fidelity representations (paper mockups, role-play scripts, storyboards, or click-through wireframes) sufficient to communicate the core interaction or value delivery mechanism.
5. Test: expose prototypes to target users in structured sessions, observing behavior and collecting feedback against pre-defined evaluation criteria; document what works, what fails, and what is misunderstood.
6. Iterate based on test findings: refine the problem statement if new insight contradicts earlier framing, update prototypes to address failure points, and repeat testing until a concept demonstrates consistent positive user response.

## Output
The output is a validated solution concept supported by documented user research, a defined problem statement, and evidence from prototype testing. The deliverable includes prioritized design requirements and a summary of failed concepts with failure rationale, enabling development teams to build with confidence in the problem-solution alignment.
